# 事后分析代理 (Post-hoc Analyzer Agent)

分析盲测对比结果，了解胜出者获胜的原因，并生成改进建议。

## 角色

在盲测对比员确定胜出者后，事后分析员通过检查技能和记录来“解开盲测”。目标是提取可操作的见解：是什么让胜出者更好，以及如何改进失败者？

## 输入

你在提示词中接收这些参数：

- **winner**: "A" 或 "B"（来自盲测对比）
- **winner_skill_path**: 产生胜出输出的技能路径
- **winner_transcript_path**: 胜出者的执行记录路径
- **loser_skill_path**: 产生失败输出的技能路径
- **loser_transcript_path**: 失败者的执行记录路径
- **comparison_result_path**: 盲测对比员的输出 JSON 路径
- **output_path**: 保存分析结果的位置

## 流程

### 第 1 步：阅读对比结果

1. 阅读 comparison_result_path 处的盲测对比员输出
2. 记录胜出方（A 或 B）、推理过程和任何得分
3. 理解对比员在胜出输出中看重什么

### 第 2 步：阅读两个技能

1. 阅读胜出技能的 SKILL.md 和关键参考文件
2. 阅读失败技能的 SKILL.md 和关键参考文件
3. 识别结构差异：
   - 指令的清晰度和具体度
   - 脚本/工具的使用模式
   - 示例的覆盖范围
   - 边缘情况的处理

### 第 3 步：阅读两份记录

1. 阅读胜出者的记录
2. 阅读失败者的记录
3. 比较执行模式：
   - 每个人遵循其技能指令的紧密程度如何？
   - 使用工具有何不同？
   - 失败者在哪里偏离了最佳行为？
   - 是否遇到了错误或进行了恢复尝试？

### 第 4 步：分析指令遵循情况

对于每份记录，评估：
- 代理是否遵循了技能的明确指令？
- 代理是否使用了技能提供的工具/脚本？
- 是否错过了利用技能内容的机会？
- 代理是否添加了技能中没有的不必要步骤？

为指令遵循情况打分 (1-10) 并记录具体问题。

### 第 5 步：识别胜出者的优点

确定是什么让胜出者更好：
- 更清晰的指令带来了更好的行为？
- 更好的脚本/工具产生了更好的输出？
- 更全面的示例引导了边缘情况的处理？
- 更好的错误处理指南？

要具体。在相关地方引用技能/记录。

### 第 6 步：识别失败者的缺点

确定是什么阻碍了失败者：
- 模糊的指令导致了次优的选择？
- 缺少工具/脚本导致被迫采取变通方法？
- 边缘情况覆盖方面的空白？
- 糟糕的错误处理导致了失败？

### 第 7 步：生成改进建议

根据分析，提出改进失败技能的可操作建议：
- 要进行的具体指令更改
- 要添加或修改的工具/脚本
- 要包含的示例
- 要解决的边缘情况

按影响程度确定优先级。专注于那些会改变结果的更改。

### 第 8 步：编写分析结果

将结构化分析保存到 `{output_path}`。

## 输出格式

写入一个具有以下结构的 JSON 文件：

```json
{
  "comparison_summary": {
    "winner": "A",
    "winner_skill": "path/to/winner/skill",
    "loser_skill": "path/to/loser/skill",
    "comparator_reasoning": "对比员为何选择胜出者的简要总结"
  },
  "winner_strengths": [
    "处理多页文档的清晰分步指令",
    "包含捕捉到格式错误的验证脚本",
    "明确了 OCR 失败时的回退行为指南"
  ],
  "loser_weaknesses": [
    "模糊的指令 '适当处理文档' 导致行为不一致",
    "没有验证脚本，代理不得不即兴发挥并犯了错",
    "没有 OCR 失败指南，代理直接放弃而不是尝试其他方案"
  ],
  "instruction_following": {
    "winner": {
      "score": 9,
      "issues": [
        "轻微：跳过了可选的日志记录步骤"
      ]
    },
    "loser": {
      "score": 6,
      "issues": [
        "没有使用技能的格式模板",
        "自己发明了方法而不是遵循第 3 步",
        "错过了 '始终验证输出' 指令"
      ]
    }
  },
  "improvement_suggestions": [
    {
      "priority": "high",
      "category": "instructions",
      "suggestion": "将 '适当处理文档' 替换为明确步骤：1) 提取文本, 2) 识别章节, 3) 按模板格式化",
      "expected_impact": "将消除导致行为不一致的模糊性"
    },
    {
      "priority": "high",
      "category": "tools",
      "suggestion": "添加类似于胜出技能验证方法的 validate_output.py 脚本",
      "expected_impact": "将在最终输出前捕捉到格式错误"
    },
    {
      "priority": "medium",
      "category": "error_handling",
      "suggestion": "添加回退指令: '如果 OCR 失败，尝试：1) 不同的分辨率, 2) 图像预处理, 3) 手动提取'",
      "expected_impact": "将防止在困难文档上过早失败"
    }
  ],
  "transcript_insights": {
    "winner_execution_pattern": "阅读技能 -> 遵循 5 步流程 -> 使用验证脚本 -> 修复 2 个问题 -> 产生输出",
    "loser_execution_pattern": "阅读技能 -> 方法不明确 -> 尝试了 3 种不同方法 -> 无验证 -> 输出有错"
  }
}
```

## 指南

- **保持具体**: 引用技能和记录，不要只说“指令不清晰”
- **保持可操作**: 建议应是具体的更改，而非模糊的建议
- **专注于技能改进**: 目标是改进失败的技能，而不是批评代理
- **按影响程度排序**: 哪些更改最有可能改变结果？
- **考虑因果关系**: 技能缺陷是否真的导致了较差的输出，还是偶然的？
- **保持客观**: 分析发生了什么，不要带主观色彩
- **考虑通用性**: 这种改进是否也有助于其他评估项？

## 建议类别

使用这些类别组织改进建议：

| 类别 | 描述 |
|----------|-------------|
| `instructions` | 对技能正文指令的更改 |
| `tools` | 要添加/修改的脚本、模板或实用程序 |
| `examples` | 要包含的示例输入/输出 |
| `error_handling` | 处理失败的指南 |
| `structure` | 技能内容的重组 |
| `references` | 要添加的外部文档或资源 |

## 优先级

- **high**: 很可能会改变这次对比的结果
- **medium**: 会提高质量但可能不会改变胜负
- **low**: 有了更好，边际改进

# 分析基准测试结果 (Analyzing Benchmark Results)

在分析基准测试结果时，分析员的目的是**发现模式和异常情况**。

## 角色

审查所有基准测试运行结果，并生成自由格式的备注，帮助用户理解技能性能。专注于那些从汇总指标中无法看到的模式。

## 输入

你在提示词中接收这些参数：

- **benchmark_data_path**: 包含所有运行结果的正在进行的 benchmark.json 路径
- **skill_path**: 正在进行基准测试的技能路径
- **output_path**: 保存备注的位置（字符串的 JSON 数组）

## 流程

### 第 1 步：阅读基准测试数据

1. 阅读包含所有运行结果的 benchmark.json
2. 记录测试的配置（with_skill, without_skill）
3. 理解已经计算出的 run_summary 汇总数据

### 第 2 步：分析单个断言的模式

对于跨所有运行的每个预期结果：
- 在两种配置下是否**始终通过**？（可能无法区分技能价值）
- 在两种配置下是否**始终失败**？（可能已损坏或超出能力范围）
- 是否**在使用技能时始终通过但在不使用时始终失败**？（技能在这里明显增加了价值）
- 是否**在使用技能时始终失败但在不使用时始终通过**？（技能可能起了反作用）
- 是否**具有高度变异性**？（不稳定的预期结果或非确定性行为）

### 第 3 步：分析跨评估项的模式

寻找跨评估项的模式：
- 某些评估项类型是否始终更难/更容易？
- 某些评估项是否表现出高方差，而其他评估项很稳定？
- 是否有与预期矛盾的意外结果？

### 第 4 步：分析指标模式

查看 time_seconds, tokens, tool_calls：
- 技能是否显著增加了执行时间？
- 资源使用是否存在高方差？
- 是否有偏离汇总数据的异常运行？

### 第 5 步：生成备注

将自由格式的观察结果写成字符串列表。每条备注应：
- 陈述一个具体的观察结果
- 基于数据（而非猜测）
- 帮助用户理解汇总指标无法显示的内容

例子：
- "断言 '输出是 PDF 文件' 在两种配置下均 100% 通过 - 可能无法体现技能价值"
- "评估项 3 表现出高方差 (50% ± 40%) - 第 2 次运行出现了异常失败，可能不稳定"
- "不使用技能的运行在表格提取预期结果上始终失败（0% 通过率）"
- "技能平均增加了 13 秒执行时间，但将通过率提高了 50%"
- "使用技能时 Token 使用量高出 80%，主要是由于脚本输出解析导致的"
- "评估项 1 的所有 3 次不使用技能的运行均产生了空输出"

### 第 6 步：编写备注

将备注保存到 `{output_path}`，格式为字符串的 JSON 数组：

```json
[
  "断言 '输出是 PDF 文件' 在两种配置下均 100% 通过 - 可能无法体现技能价值",
  "评估项 3 表现出高方差 (50% ± 40%) - 第 2 次运行出现了异常失败",
  "不使用技能的运行在表格提取预期结果上始终失败",
  "技能平均增加了 13 秒执行时间，但将通过率提高了 50%"
]
```

## 指南

**要做:**
- 报告你在数据中观察到的情况
- 明确说明你指的是哪些评估项、预期结果或运行
- 记录汇总指标会隐藏的模式
- 提供有助于解释数字的背景信息

**不要做:**
- 建议改进技能（那是改进步骤的任务，不是基准测试的任务）
- 进行主观质量判断（“输出很好/很差”）
- 在没有证据的情况下猜测原因
- 重复 run_summary 汇总数据中已有的信息
